## TL;DR

- Interfaces are **technical contracts** that live on a **node**, not on a relationship.
- CALM 1.2 has two interface forms: freeform (`interface-type`) and formal (`interface-definition` with `definition-url` + `config`).
- The old `url-interface` and `host-port-interface` types from 1.0-rc1 are gone in 1.2 — do not copy them from old examples.
- Relationships reference interfaces via `source.interfaces` and `destination.interfaces` — arrays of interface `unique-id` strings.
- Interfaces enable automated tooling: port scanning, schema validation, and security surface mapping.

## Why it matters

A `connects` relationship tells you that node A communicates with node B. That is useful, but it does not tell you *how*. Is it HTTP? HTTPS? gRPC? On which port? With what schema? Interfaces answer those questions.

More than documentation, interfaces are machine-readable contracts. CALM Guard can compare an interface's declared `protocol` against a `mutual-tls` control and flag non-compliance. A network scanner can verify that the declared port is actually open. A schema validator can confirm that the payload matches the declared OpenAPI spec. None of that is possible without the interface.

If you want CALM to be more than a diagram, interfaces are where you start adding the precision that makes it so.

## The concept

### Interface vs relationship — the distinction

The distinction is direct: **interfaces describe where and how to reach a node; relationships describe whether two nodes are connected**.

| Thing | Where it lives | What it answers |
|---|---|---|
| Interface | On a **node** | How do you reach me? (port, protocol, schema, URL) |
| Relationship | In the `relationships[]` array | Are these two nodes connected? Via which mechanism? |

A `connects` relationship can exist with no interface references — it just says "these nodes talk". Add interfaces and the relationship becomes auditable: "they talk, and we know exactly which interface on each side participates in the connection."

### The freeform form (interface-type)

The freeform interface form is a JSON object with `unique-id` plus any additional properties you choose. The schema allows `additionalProperties: true` for this form. Use it when you do not have a canonical external schema to reference, or when the protocol is simple enough that free-form fields suffice.

```json
{
  "unique-id": "orders-api-rest",
  "name": "Orders REST API",
  "protocol": "HTTPS",
  "port": 8443
}
```

You can add whatever fields make the interface useful: `url`, `schema`, `description`, `specification`, `operations`. The only field required by the schema is `unique-id`.

Common additional fields you will see in production CALM files:

```json
{
  "unique-id": "trade-feed-websocket",
  "name": "Trade Feed WebSocket API",
  "protocol": "WebSocket",
  "port": 18086,
  "schema": {
    "specification": "SocketIO",
    "operations": [
      { "event": "subscribe", "direction": "inbound" },
      { "event": "publish", "direction": "outbound" }
    ]
  }
}
```

Use freeform interfaces in most cases. They are easy to read, easy to extend, and sufficient for the majority of architectural documentation needs.

### The formal form (interface-definition)

The formal interface form uses `definition-url` to reference an external JSON Schema and `config` to provide the configuration that satisfies that schema. Use this when you need machine-validated, cross-team-compatible interface definitions — for example, when your platform team has published canonical schema definitions that all services must conform to.

```json
{
  "unique-id": "orders-api-host-port",
  "definition-url": "https://calm.finos.org/release/1.2/prototype/interface/tcp-host-port.json",
  "config": {
    "host": "orders-api.internal",
    "port": 8443
  }
}
```

All three fields are required for the formal form: `unique-id`, `definition-url`, and `config`. The schema does not allow `additionalProperties` on this form — the shape is strictly governed by the external schema referenced at `definition-url`.

FINOS provides example interface schemas for common patterns:

| Schema | What it describes |
|---|---|
| `calm/interfaces/example/url.json` | A service reachable by URL |
| `calm/interfaces/example/tcp-host-port.json` | A TCP endpoint (host + port) |
| `calm/interfaces/example/tcp-port.json` | A port-only TCP interface |
| `calm/interfaces/example/container-image.json` | A container image interface |

When to use the formal form:

- Your organisation has published internal canonical interface schemas
- You are contributing to a CALM Pattern that must be instantiated by downstream architectures
- You want a CALM Guard rule to verify that the `config` matches the referenced schema
- You are federating architectures across teams and need a common interface vocabulary

### How relationships reference interfaces

An interface living on a node is not automatically "activated" by every relationship involving that node. You must explicitly reference it from the relationship using `source.interfaces` and `destination.interfaces` — both are arrays of interface `unique-id` strings.

```json
{
  "unique-id": "portal-to-orders-api",
  "description": "Analyst portal calls the Orders API over HTTPS.",
  "relationship-type": {
    "connects": {
      "source": {
        "node": "analyst-portal",
        "interfaces": ["analyst-portal-url"]
      },
      "destination": {
        "node": "orders-api",
        "interfaces": ["orders-api-rest"]
      }
    }
  },
  "protocol": "HTTPS"
}
```

Why arrays? A node can expose multiple interfaces (REST API + WebSocket + health-check endpoint). A single `connects` relationship might involve more than one of them — for example, a load balancer that connects via both a primary HTTPS interface and a separate health-check port. The arrays let you be specific.

If you do not provide `source.interfaces` or `destination.interfaces`, the relationship still validates — but downstream tooling cannot determine which specific interface is involved.

### Why 1.0-rc1 interface types are gone

CALM 1.0-rc1 had built-in interface type definitions with string constants: `url-interface`, `host-port-interface`, `container-image-interface`, and `port-interface`. These were embedded directly in the CALM schema as named types.

In CALM 1.1+, this approach was replaced with the more flexible `definition-url` pattern. Instead of baking interface types into the CALM core schema, you reference external schemas. This allows the interface library to grow without modifying the CALM core spec, and allows organisations to publish their own interface schemas.

If you copy a pre-1.1 CALM example, you will encounter:

```json
{
  "$ref": "https://calm.finos.org/release/1.0-rc1/meta/interface.json#/defs/url-interface"
}
```

This does not exist in CALM 1.2. Your file will fail `calm validate`. Replace these with either the freeform form or the `definition-url` form using the FINOS 1.2 example schemas.

## Common mistakes

- **Putting interface details on the relationship.** The `connects` relationship already has `protocol` at the top level. If you find yourself adding `port` or `url` to the relationship sub-object rather than to a node's `interfaces[]`, stop. Interfaces live on the node.
- **Using `url-interface` from 1.0-rc1 in a 1.2 file.** If your file has `"$ref": "...1.0-rc1/meta/interface.json#/defs/url-interface"`, it will fail CALM CLI 1.2 validation. Replace it with a freeform interface-type object or a 1.2 `definition-url` reference.
- **Confusing `protocol` on the relationship vs `protocol` on the interface.** The `protocol` on a `connects` relationship (e.g. `"protocol": "HTTPS"`) describes the wire protocol for that specific connection. The `protocol` field inside an interface object describes the protocol that interface supports. Both can exist and they should agree, but they are not the same field.
- **Forgetting `unique-id` on the interface object.** Every interface must have `unique-id`. Without it, you cannot reference the interface from a relationship's `source.interfaces` or `destination.interfaces` array, and the file will fail schema validation.
- **Naming two interfaces the same `unique-id` on the same node.** Interface unique-ids must be unique within the architecture (not just within the node). If two different nodes both have an interface with the same `unique-id`, you will get unexpected behavior when relationships try to reference them.
- **Omitting `config` from a formal interface-definition.** The formal form requires exactly three fields: `unique-id`, `definition-url`, and `config`. If you omit `config`, validation fails even if `definition-url` is present.
