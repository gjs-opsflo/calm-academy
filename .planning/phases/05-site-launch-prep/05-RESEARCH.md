## Summary

Phase 5 scaffolds and ships the Docusaurus site. The core pattern is: Docusaurus 3.10.1 installed in `site/`, content wired via plugin `path: '../content'` (sibling directory — works in Docusaurus 3), images served from `site/static/img/` (symlinked to `../../illustrations/exported/`), a custom `Quiz.tsx` component importing YAML via `js-yaml`, and a GitHub Actions native Pages workflow.

**Two critical findings discovered in the codebase require attention before the site can build:**

1. **Image path inconsistency:** Modules 0–1 use `../../illustrations/exported/filename.svg` (correct relative path from their position) while modules 2–3 use `../illustrations/exported/filename.svg` (this path resolves to `content/illustrations/exported/` which does not exist — broken). The Docusaurus `static/img/` symlink approach only works if MDX files use `/img/filename.svg` absolute paths OR if the image paths are corrected. The plan must include a content-fix task.

2. **Empty duplicate module directories:** `content/` contains both `module-02-calm-fundamentals/` (7 files, canonical) and `module-02-fundamentals/` (empty), plus `module-03-calm-ecosystem/` (7 files, canonical) and `module-03-ecosystem/` (empty). The empty directories should be removed to prevent autogenerated sidebar from generating ghost categories. The plan must include a cleanup task.

**Primary recommendation:** Use `docs: { path: '../content' }` in `docusaurus.config.js` (not symlinks for the docs source — Docusaurus has known issues with symlinked docs directories). Keep the `static/img/` symlink for illustrations and update MDX image references to use `/img/<filename>.svg` absolute paths. Add a pre-build step to the CI workflow that verifies image symlink is in place.

### Recommended Project Structure

```
site/
├── docs/                    # NOT a symlink — use path: '../content' in config instead
├── static/
│   ├── img/                 # Symlink → ../../illustrations/exported/ (WORKS for static assets)
│   └── .nojekyll            # Required — prevents Jekyll processing on GitHub Pages
├── src/
│   ├── components/
│   │   └── Quiz.tsx         # Custom quiz component
│   └── css/
│       └── custom.css       # Minimal overrides (CALM Academy title only)
├── docusaurus.config.js     # Site configuration
├── sidebars.js              # Autogenerated sidebar config
└── package.json             # Docusaurus + js-yaml deps
```

**Note on symlinks vs path config:** The `site/docs/` directory should NOT be a symlink. Instead, use `docs: { path: '../content' }` in `docusaurus.config.js`. The Docusaurus docs plugin accepts relative paths to sibling directories (outside `site/`) and this is a standard monorepo pattern. [CITED: Docusaurus issue discussion — path: '../other-section-docs' shown as working example]

Symlinks to the `docs` directory have a separate well-documented issue history in Docusaurus (issues #3272, #6257, #10751) — they produce cryptic errors and are effectively unsupported. The `static/img/` symlink for illustration SVGs is unaffected by this problem because static assets are copied verbatim without MDX processing.

### Pattern 3: Quiz.tsx Component

The quiz YAML schema (from `docs-meta/QUIZ-FORMAT.md`) has these types: `multiple_choice`, `multiple_correct`, `short_answer`, `code_completion`. The Quiz component needs to handle at minimum `multiple_choice` and `short_answer` for launch (the 4 existing quiz files use primarily these types).

```tsx
// site/src/components/Quiz.tsx
// Data contract derived from quizzes/module-NN-*.yaml schema

import React, { useState } from 'react';
import yaml from 'js-yaml';

// Quiz YAML is imported at build time via standard import
// The YAML files live at quizzes/ (repo root) — one level above site/
// Import path from Quiz.tsx (at site/src/components/): ../../../quizzes/<file>
// This requires the 'file' prop to map to the correct import

// RECOMMENDED APPROACH: Copy quiz YAML to site/static/quizzes/ at build time
// via a prebuild script OR use require() with dynamic path

interface QuizOption {
  label: string;
  correct: boolean;
  explanation: string;
}

interface QuizQuestion {
  id: string;
  type: 'multiple_choice' | 'multiple_correct' | 'short_answer' | 'code_completion';
  prompt: string;
  options?: QuizOption[];
  accepted_answers?: string[];
  case_sensitive?: boolean;
  explanation?: string;
  reference_section?: string;
}

interface QuizChapter {
  chapter: number;
  slug: string;
  title: string;
  questions: QuizQuestion[];
}

interface QuizData {
  module: number;
  title: string;
  chapters: QuizChapter[];
}

// Component receives parsed JS object (not raw YAML string)
interface QuizProps {
  data: QuizData;  // caller imports the YAML and passes parsed object
}

export default function Quiz({ data }: QuizProps) {
  // ... submit-all grading, score display, per-question explanations
}
```

**YAML import strategy:** The cleanest approach for Docusaurus is to pre-copy `quizzes/*.yaml` files to `site/static/quizzes/` (or `site/src/quizzes/`) as part of the build process, then import them in MDX files using:

```mdx
import quizData from '../../quizzes/module-00-on-ramp.yaml';
import Quiz from '@site/src/components/Quiz';

<Quiz data={quizData} />
```

This requires adding a Webpack yaml-loader rule via `docusaurus.config.js`'s `webpack` field, OR simply converting YAML to JSON at prebuild time. The simplest approach for a static site is a **prebuild script** that runs `js-yaml` to convert each quiz YAML to a JSON file importable without a webpack loader.

### Pattern 4: GitHub Actions Native Pages Workflow

```yaml
# .github/workflows/site-deploy.yml
# Source: https://docusaurus.io/docs/deployment#deploying-to-github-pages
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    name: Build Docusaurus
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-node@v4
        with:
          node-version: 22          # Node 20 is EOL as of April 2026; use Node 22
          cache: npm
          cache-dependency-path: site/package-lock.json
      - name: Install site dependencies
        run: cd site && npm ci
      - name: Build site
        run: cd site && npm run build
      - name: Upload Pages artifact
        if: github.ref == 'refs/heads/main'
        uses: actions/upload-pages-artifact@v3
        with:
          path: site/build

  deploy:
    name: Deploy to GitHub Pages
    needs: build
    if: github.ref == 'refs/heads/main'
    permissions:
      pages: write
      id-token: write
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

**Required:** Add `site/static/.nojekyll` file (empty) to prevent GitHub Pages from running Jekyll on the build output. Without this, Docusaurus-generated paths with underscores are dropped.

[CITED: https://docusaurus.io/docs/deployment#deploying-to-github-pages]

### Anti-Patterns to Avoid

- **Symlinking `site/docs/` to `../content/`:** Has well-documented bugs in Docusaurus (#3272, #6257, #10751). Use `docs: { path: '../content' }` in config instead.
- **Importing YAML directly without a webpack loader:** Standard `import data from '../../quizzes/foo.yaml'` fails because Webpack does not parse YAML by default. Either add yaml-loader config or convert YAML to JSON at prebuild.
- **Using `/img/...` absolute paths before verifying `static/img/` symlink works in CI:** Linux runners follow symlinks; verify with a build test before relying on this.
- **Omitting `site/static/.nojekyll`:** GitHub Pages strips files/directories starting with `_` when Jekyll processing is active. Docusaurus output has `_` prefixed directories.
- **Using Node 20 in CI:** Node 20 reached EOL on April 30, 2026. The existing `ci.yml` already uses Node 22. Use Node 22 for consistency.
- **Leaving placeholder Algolia config active for production:** The `appId: 'PLACEHOLDER'` config causes a visible broken search bar. Either remove the algolia key entirely until approved, or use `@easyops-cn/docusaurus-search-local` as the active plugin.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Static site with MDX support | Custom MDX pipeline | Docusaurus | Hot reload, broken link checking, sidebar, SEO, plugins |
| Client-side search | Custom search index | `@easyops-cn/docusaurus-search-local` | Handles tokenization, stopwords, multi-field indexing |
| YAML parsing | Custom parser | `js-yaml` | 100M+ weekly downloads; handles all YAML 1.2 edge cases |
| GitHub Pages deploy | `gh-pages` npm branch approach | GitHub Actions native Pages workflow | No `gh-pages` branch churn; artifact-based; faster; CONTEXT.md locked |
| Sidebar navigation | Hand-maintained `sidebars.js` | Autogenerated | Directory structure IS the sidebar; updating is zero-maintenance |

**Key insight:** Docusaurus eliminates the "glue" layer that usually breaks static site deployments. The build step, broken link detection, image handling, and sidebar generation are all built in — resisting the urge to customize them saves weeks of maintenance.

## Critical Findings (Action Required in Plan)

### Critical Finding #1: Image Path Bug in Modules 02 and 03

**What:** MDX files in `content/module-02-calm-fundamentals/` and `content/module-03-calm-ecosystem/` reference images as `../illustrations/exported/<filename>.svg`. From those directories, `../` resolves to `content/` — but `content/illustrations/` does not exist. These image links are currently broken.

By contrast, `content/module-00-on-ramp/` and `content/module-01-case-for-aac/` correctly use `../../illustrations/exported/<filename>.svg`, which resolves to `calm-academy/illustrations/exported/` (the actual location).

**Affected files:** All 14 MDX files in module-02 and module-03 that embed illustrations — verified by grep.

**Required fix:** One of two approaches:
- **Approach A (recommended):** Update all MDX image references to use Docusaurus's recommended absolute `/img/<filename>.svg` path, which resolves via `site/static/img/` (the symlinked directory). This is the cleanest fix and works regardless of content directory depth.
- **Approach B (alternative):** Fix the relative paths in module-02/03 to use `../../illustrations/exported/` (matching module-00/01). This works when serving locally but does NOT work when Docusaurus processes them (images must be either in `static/` or imported as modules).

**Recommendation: Approach A.** Use `/img/<filename>.svg` in all MDX files. This is the Docusaurus-canonical approach. [CITED: https://docusaurus.io/docs/static-assets — "use absolute paths in standard Markdown syntax"]

Total affected image references across module-02 and module-03: approximately 23 occurrences across 14 files.

### Critical Finding #2: Empty Duplicate Module Directories

**What:** `content/` contains empty placeholder directories that will produce ghost sidebar entries in Docusaurus:
- `content/module-02-fundamentals/` — empty (canonical: `module-02-calm-fundamentals/`)
- `content/module-03-ecosystem/` — empty (canonical: `module-03-calm-ecosystem/`)
- `content/module-04-governance/` — empty (not yet authored; expected to be out of scope)
- `content/module-05-ai-native/` — empty
- `content/module-06-enterprise/` — empty

**Required fix:** Remove the empty `module-02-fundamentals/` and `module-03-ecosystem/` directories (they are naming artifacts from bootstrap phase). The others (`module-04/` through `module-06/`) can either be removed or excluded from the sidebar via `_category_.json` with `"hidden": true`.

### Critical Finding #3: Node Version Mismatch

**What:** CONTEXT.md specifies Node 20 LTS, but Node 20 reached end-of-life on April 30, 2026 (it is now June 2026). The existing `ci.yml` already uses Node 22. The new `site-deploy.yml` must use Node 22 for consistency and security.

## Common Pitfalls

### Pitfall 1: Docusaurus Symlink Docs Directory
**What goes wrong:** Creating `site/docs/` as a symlink to `../content/` causes Docusaurus build to fail with cryptic errors ("Cannot read properties of undefined (reading 'id')") or pagination link failures.
**Why it happens:** Docusaurus internally resolves symlinks to real paths, which can create path mismatches in the metadata pipeline.
**How to avoid:** Use `docs: { path: '../content' }` in `docusaurus.config.js` instead of a symlink. This is the supported pattern for sibling-directory content sources.
**Warning signs:** Build fails on `npm run build` even though `npm start` (dev mode) may partially work.

### Pitfall 2: Missing `.nojekyll` File
**What goes wrong:** GitHub Pages drops directories and files starting with `_` (e.g., `_next/`, `_docusaurus/`). Docusaurus build output uses these paths. Result: site deploys but loads completely blank.
**Why it happens:** GitHub Pages default Jekyll processing strips underscore-prefixed paths.
**How to avoid:** Add empty `site/static/.nojekyll` file. Docusaurus copies it to build root. Some Docusaurus versions add it automatically — add explicitly to be safe.
**Warning signs:** Site deploys (green workflow) but renders blank page or 404 in browser.

### Pitfall 3: YAML Import Without Webpack Loader
**What goes wrong:** MDX file uses `import data from '../../../quizzes/module-00.yaml'`; Webpack throws "Module parse failed" because it doesn't know how to handle YAML.
**Why it happens:** Webpack requires explicit loaders for non-JS/JSON file types. Docusaurus does not ship a YAML loader by default.
**How to avoid:** Convert YAML to JSON at prebuild time (simplest), OR add `yaml-loader` via `docusaurus.config.js` `webpack` field. The prebuild approach requires no extra npm package and is more transparent.
**Warning signs:** Build fails with "Module parse failed: Unexpected token" pointing to a `.yaml` file.

### Pitfall 4: `baseUrl` Not Applied to Image Paths
**What goes wrong:** Images at `/img/filename.svg` render in `npm start` (baseUrl is `/`) but 404 in production (baseUrl is `/calm-academy/`).
**Why it happens:** When using Markdown image syntax `![alt](/img/file.svg)`, Docusaurus rewrites these paths to include `baseUrl` automatically. If the image is referenced from JSX (inside Quiz.tsx), you must use `useBaseUrl('/img/file.svg')` from `@docusaurus/useBaseUrl` to get the correct path.
**How to avoid:** In MDX/Markdown: use `/img/filename.svg` (Docusaurus rewrites automatically). In TSX/JSX components: use `import useBaseUrl from '@docusaurus/useBaseUrl'; const imgUrl = useBaseUrl('/img/file.svg');`
**Warning signs:** Images work in dev, 404 in production deployment at `/calm-academy/`.

### Pitfall 5: Algolia Placeholder Showing as Broken Search Bar
**What goes wrong:** `themeConfig.algolia` with `appId: 'PLACEHOLDER'` renders a search bar that fails silently or shows an error when users try to search.
**Why it happens:** The Algolia plugin initializes on page load and makes network requests to the configured appId.
**How to avoid:** Do NOT include the `algolia` key in `themeConfig` until a real appId is obtained. Separately add `@easyops-cn/docusaurus-search-local` as an active `themes` entry for dev/pre-launch search.
**Warning signs:** Search bar appears but returns no results or throws a console error about Algolia.

### Pitfall 6: Autogenerated Sidebar Ordering
**What goes wrong:** Module chapters appear in random/alphabetical order instead of chapter order (e.g., `building-your-first-architecture.mdx` sorts before `calm-specification.mdx` alphabetically, but chapter order is `calm-specification → nodes → relationships → interfaces → controls → decorators → building`).
**Why it happens:** Autogenerated sidebar sorts by filename alphabetically when no `sidebar_position` frontmatter is present. The content filenames don't have numeric prefixes.
**How to avoid:** Add `sidebar_position` to each MDX file's frontmatter OR add `_category_.json` to each module directory with positions. Adding `sidebar_position` to frontmatter is the lowest-friction approach since files already have `chapter: N.M` in frontmatter — the executor can map `chapter: 3.1` → `sidebar_position: 1`.
**Warning signs:** Sidebar shows chapters out of pedagogical order.

### Quiz YAML → JSON prebuild conversion (Node script)
```javascript
// scripts/convert-quiz-yaml.mjs
// Run before Docusaurus build to make quiz YAML importable
import { readFileSync, writeFileSync, readdirSync } from 'fs';
import { load } from 'js-yaml';
import { join } from 'path';

const quizDir = 'quizzes';
const outputDir = 'site/src/quizzes';

for (const file of readdirSync(quizDir)) {
  if (!file.endsWith('.yaml')) continue;
  const data = load(readFileSync(join(quizDir, file), 'utf8'));
  writeFileSync(
    join(outputDir, file.replace('.yaml', '.json')),
    JSON.stringify(data, null, 2)
  );
}
```

## Open Questions

1. **Does `docs: { path: '../content' }` work in Docusaurus CI (Linux runner)?**
   - What we know: Sibling-directory paths (`../other-section-docs`) are cited as supported in Docusaurus GitHub issues and monorepo discussions. The restriction is only on using a path that contains the site directory itself (i.e., path resolves to `..` which is the parent).
   - What's unclear: Whether the CI checkout gives Docusaurus enough filesystem context to resolve `../content` relative to the `site/` working directory.
   - Recommendation: Add a build-site test in Wave 1 before committing to this approach. If it fails, the fallback is a prebuild `cp -r content site/docs` step.

2. **Should the `static/img/` symlink work on GitHub Actions Ubuntu runner?**
   - What we know: Linux supports symlinks natively. GitHub Actions `actions/checkout@v4` preserves symlinks tracked in git.
   - What's unclear: Whether the symlink resolves correctly in CI when `site/static/img/` → `../../illustrations/exported/`.
   - Recommendation: Track the symlink in git (`git add site/static/img`). The CI runner will follow it at build time. Verify in first build test.

3. **Quiz YAML → JSON prebuild vs webpack yaml-loader?**
   - What we know: Both approaches work. Prebuild is simpler (no new npm package, no webpack config). yaml-loader gives a more "native" import experience.
   - Recommendation: Use prebuild JSON conversion for launch simplicity. [ASSUMED]
