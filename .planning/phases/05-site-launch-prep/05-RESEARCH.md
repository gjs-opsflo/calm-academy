## Summary

Phase 5 scaffolds and ships the Docusaurus site. The core pattern is: Docusaurus 3.10.1 installed in `site/`, content wired via `docs: { path: '../content' }` in `docusaurus.config.js` (sibling directory — the supported Docusaurus pattern), images served from `site/static/img/` (symlinked to `../../illustrations/exported/`), a custom `Quiz.tsx` component importing pre-converted JSON quiz data, and a GitHub Actions native Pages workflow.

**Three critical findings discovered in the codebase require attention before the site can build:**

1. **Image path bug:** Modules 02 and 03 use `../illustrations/exported/<file>.svg` (broken — resolves to `content/illustrations/`). Modules 00 and 01 use `../../illustrations/exported/<file>.svg` (also broken in Docusaurus — must be absolute `/img/<file>.svg`). All 4 modules need image path updates to Docusaurus-canonical absolute paths.

2. **Empty duplicate module directories:** `content/module-02-fundamentals/` and `content/module-03-ecosystem/` are empty directories (naming artifacts from bootstrap). They will produce ghost sidebar entries. Directories `module-04-governance/`, `module-05-ai-native/`, and `module-06-enterprise/` also exist as empty placeholders. All must be removed or excluded before site build.

3. **Node version:** CONTEXT.md specifies Node 20 LTS, but Node 20 is EOL as of April 30, 2026. The existing `ci.yml` already uses Node 22. The `site-deploy.yml` must use Node 22 for consistency. [VERIFIED: npm registry — existing ci.yml uses node-version: '22']

**Two additional codebase facts discovered during research:**

4. **No `sidebar_position` frontmatter exists** in any MDX file — all files use chapter numbers in frontmatter (`chapter: 0.1`) but not the `sidebar_position` key. Sidebar ordering requires adding `_category_.json` per module directory and `sidebar_position` to each MDX file's frontmatter.

5. **Quiz YAML filename discrepancy:** The quiz file for Module 3 is `quizzes/module-03-ecosystem.yaml` (not `module-03-calm-ecosystem.yaml`). Quiz component and MDX usage must reference this exact filename. [VERIFIED: codebase inspection]

**Primary recommendation:** Use `docs: { path: '../content' }` in `docusaurus.config.js` (not symlinks for the docs source). Update all 23 broken image references across modules 02 and 03 to use `/img/<filename>.svg` absolute paths; update modules 00 and 01 image paths from `../../illustrations/exported/` to `/img/<filename>.svg` as well. Use a prebuild script to convert quiz YAML to JSON. Remove empty placeholder directories before first build.

**Note on symlinks vs path config:** The `site/docs/` directory must NOT be a symlink. Use `docs: { path: '../content' }` in `docusaurus.config.js`. ### Pattern 4: Quiz YAML → JSON Prebuild

The simplest approach for Docusaurus is to convert quiz YAML to JSON before build. This avoids webpack loader configuration and makes quiz data importable as standard JSON modules.

```javascript
// scripts/convert-quiz-yaml.mjs
// Run before Docusaurus build to make quiz YAML importable as JSON
import { readFileSync, writeFileSync, mkdirSync, readdirSync } from 'fs';
import { load } from 'js-yaml';
import { join } from 'path';

const quizDir = 'quizzes';
const outputDir = 'site/src/quizzes';

mkdirSync(outputDir, { recursive: true });

for (const file of readdirSync(quizDir)) {
  if (!file.endsWith('.yaml')) continue;
  const data = load(readFileSync(join(quizDir, file), 'utf8'));
  writeFileSync(
    join(outputDir, file.replace('.yaml', '.json')),
    JSON.stringify(data, null, 2)
  );
}
console.log('Quiz YAML → JSON conversion complete.');
```

Then in MDX:
```mdx
import quizData from '@site/src/quizzes/module-00-on-ramp.json';
import Quiz from '@site/src/components/Quiz';

<Quiz data={quizData} />
```

          path: site/build

    name: Deploy to GitHub Pages
    needs: build
    if: github.ref == 'refs/heads/main'
    permissions:
      pages: write
      id-token: write
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

      - name: Deploy to GitHub Pages

### Anti-Patterns to Avoid

- **Symlinking `site/docs/` to `../content/`:** Has well-documented bugs in Docusaurus (#3272, #6257, #10751). Use `docs: { path: '../content' }` in config instead.
- **Including `algolia` in `themeConfig` without real credentials:** Causes a broken search bar that makes network calls on every page load. Remove the `algolia` key entirely until DocSearch is approved; use `@easyops-cn/docusaurus-search-local` as the active search plugin.
- **Importing YAML directly without a webpack loader:** `import data from '../../../quizzes/foo.yaml'` fails. Use the prebuild JSON conversion approach.
- **Omitting `site/static/.nojekyll`:** GitHub Pages strips `_`-prefixed paths (Docusaurus output uses them extensively). Result: site deploys with green workflow but renders blank.
- **Using Node 20 in CI:** EOL April 30, 2026. The existing `ci.yml` already uses Node 22. Match it.
- **Referencing quiz as `module-03-calm-ecosystem.yaml`:** The actual filename is `module-03-ecosystem.yaml`. Using the wrong name causes a runtime 404 or import error.
- **Leaving placeholder module directories:** `content/module-02-fundamentals/`, `content/module-03-ecosystem/`, `module-04-governance/`, `module-05-ai-native/`, `module-06-enterprise/` are all empty — they produce ghost sidebar categories. Remove or hide before first build.
- **Using `/img/` paths in JSX components without `useBaseUrl`:** Images in Markdown/MDX are rewritten automatically by Docusaurus. Images referenced in TSX (e.g., inside `Quiz.tsx`) must use `import useBaseUrl from '@docusaurus/useBaseUrl'; const src = useBaseUrl('/img/file.svg')` to get the correct production URL.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Static site with MDX support | Custom MDX pipeline | Docusaurus | Hot reload, broken link checking, sidebar, SEO, plugins |
| Client-side search | Custom search index | `@easyops-cn/docusaurus-search-local` | Handles tokenization, stopwords, multi-field indexing |
| YAML parsing | Custom parser | `js-yaml` | 100M+ weekly downloads; handles all YAML 1.2 edge cases |
| GitHub Pages deploy | `gh-pages` npm branch approach | GitHub Actions native Pages workflow | No `gh-pages` branch churn; artifact-based; faster; CONTEXT.md locked |
| Sidebar navigation | Hand-maintained `sidebars.js` | Autogenerated | Directory structure IS the sidebar; updating is zero-maintenance as content grows |
| Quiz state management | Custom state library | React `useState` | Quiz state is local and transient; no global state manager needed |

**Key insight:** Docusaurus eliminates the "glue" layer that usually breaks static site deployments. The build step, broken link detection, image handling, and sidebar generation are all built in — resisting the urge to customize them saves weeks of maintenance.

## Critical Findings (Action Required in Plan)

### Critical Finding #1: Image Path Bug in ALL Modules

## Common Pitfalls

### Pitfall 1: Docusaurus Symlink Docs Directory
**What goes wrong:** Creating `site/docs/` as a symlink to `../content/` causes Docusaurus build to fail with cryptic errors ("Cannot read properties of undefined (reading 'id')") or pagination link failures.
**Why it happens:** Docusaurus internally resolves symlinks to real paths, creating path mismatches in the metadata pipeline.
**How to avoid:** Use `docs: { path: '../content' }` in `docusaurus.config.js`. This is the supported pattern for sibling-directory content sources.
**Warning signs:** Build fails on `npm run build` even though `npm start` (dev mode) may partially work.

### Pitfall 2: Missing `.nojekyll` File
**What goes wrong:** GitHub Pages drops directories and files starting with `_` (e.g., `_next/`, `_docusaurus/`). Docusaurus build output uses these paths. Result: site deploys but loads completely blank.
**Why it happens:** GitHub Pages default Jekyll processing strips underscore-prefixed paths.
**How to avoid:** Add empty `site/static/.nojekyll` file. Docusaurus copies it to build root.
**Warning signs:** Site deploys (green workflow) but renders blank page or 404 in browser.

### Pitfall 3: YAML Import Without Webpack Loader
**What goes wrong:** MDX file uses `import data from '../../../quizzes/module-00.yaml'`; Webpack throws "Module parse failed" because it doesn't know how to handle YAML.
**Why it happens:** Webpack requires explicit loaders for non-JS/JSON file types. Docusaurus does not ship a YAML loader by default.
**How to avoid:** Use the prebuild JSON conversion script. Run `node scripts/convert-quiz-yaml.mjs` before `npm run build`.
**Warning signs:** Build fails with "Module parse failed: Unexpected token" pointing to a `.yaml` file.

### Pitfall 4: `baseUrl` Not Applied to Image Paths in JSX
**What goes wrong:** Images at `/img/filename.svg` render in `npm start` (baseUrl is `/`) but 404 in production (baseUrl is `/calm-academy/`).
**Why it happens:** Docusaurus rewrites `/img/` paths in Markdown/MDX automatically. In TSX components (like `Quiz.tsx`), you must use `useBaseUrl` manually.
**How to avoid:** In MDX/Markdown: use `/img/filename.svg` (auto-rewritten). In TSX/JSX: `import useBaseUrl from '@docusaurus/useBaseUrl'; const src = useBaseUrl('/img/file.svg');`
**Warning signs:** Images work in dev, 404 in production deployment.

### Pitfall 5: Algolia Placeholder Breaking Search Bar
**What goes wrong:** `themeConfig.algolia` with `appId: 'PLACEHOLDER'` renders a search bar that fails on every page load.
**Why it happens:** The Algolia plugin initializes on page load and makes network requests to the configured appId.
**How to avoid:** Do NOT include the `algolia` key in `themeConfig` until a real appId is obtained. Use `@easyops-cn/docusaurus-search-local` via the `themes` array instead.
**Warning signs:** Search bar appears but returns no results or throws a console error about Algolia.

### Pitfall 6: Autogenerated Sidebar Alphabetical Order
**What goes wrong:** Module chapters appear in alphabetical order instead of pedagogical chapter order.
**Why it happens:** Autogenerated sidebar sorts by filename alphabetically when no `sidebar_position` frontmatter is present. The content filenames don't have numeric prefixes.
**How to avoid:** Add `sidebar_position: N` to each MDX file's frontmatter AND add `_category_.json` to each module directory with `position` and `label`.
**Warning signs:** Sidebar shows chapters out of pedagogical order (e.g., "building-your-first-architecture" appears first in Module 2 instead of last).

### Pitfall 7: Wrong Quiz Filename for Module 3
**What goes wrong:** Using `<Quiz data={quizData} />` where `quizData` was imported from `module-03-calm-ecosystem.json` — file does not exist; import fails.
**Why it happens:** Naming inconsistency — the quiz file uses the short slug `module-03-ecosystem.yaml`, not the long slug `module-03-calm-ecosystem.yaml`.
**How to avoid:** Use `module-03-ecosystem.json` as the import target (after prebuild conversion).
**Warning signs:** Build error "Cannot find module '@site/src/quizzes/module-03-calm-ecosystem.json'".

## Open Questions

1. **Does `docs: { path: '../content' }` work in Docusaurus CI (Linux runner)?**
   - What we know: Sibling-directory paths are cited as supported in Docusaurus GitHub issue discussions. The restriction is only on using a path that resolves to the site directory's ancestor.
   - What's unclear: Whether the CI checkout gives Docusaurus enough filesystem context to resolve `../content` relative to the `site/` working directory during `npm run build`.
   - Recommendation: Validate with a build-site test in Wave 1 before committing. If it fails, the fallback is `cp -r ../content docs` as a prebuild step (which conflicts with the CONTEXT.md no-copy constraint — would need user sign-off).

   - What we know: Linux supports symlinks natively. `actions/checkout@v4` preserves git-tracked symlinks.
   - What's unclear: Whether tracking a symlink that points outside the git working tree (`site/static/img` → `../../illustrations/exported`) causes issues.
   - Recommendation: Track the symlink in git (`git add site/static/img`). Verify in first build test. Fallback: add `ln -s` step to workflow.

3. **GitHub Pages environment enabled in repo settings?**
   - What we know: Requires one-time manual configuration in repo settings → Pages → Source: GitHub Actions.
   - What's unclear: Whether this has been done for `gjs-opsflo/calm-academy`.
