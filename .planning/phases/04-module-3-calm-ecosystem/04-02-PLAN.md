    Content arc:
    - TL;DR (4-5 bullets): Hub is the "npm registry for architectures" — versioned, namespaced, discoverable; `calm hub push/pull/list` ships in CLI 1.45.0; two storage modes (MongoDB for production, NitriteDB embedded for dev); Hub has a REST API AND an MCP endpoint for AI queries; Lab 3 does NOT require Hub — it's taught conceptually here.
    - Why it matters: files on disk don't scale. When 50 teams share architecture patterns, you need versioning, access control, and discoverability. Hub provides all three.
    - The concept H3 subsections:

      ### Why a registry exists
      Pain without Hub: architecture files in git repos → teams copy-paste → diverge → no single source of truth. Hub solves: every architecture and pattern has a namespace, version, and owner. Hub is to CALM architectures what Artifactory is to JARs or npm registry is to packages.

      ### Hub architecture
      Namespace-scoped resource tree. Each namespace (e.g., `org.mybank`) contains: architectures, patterns, standards, controls. Each resource has a version (semver). Two storage modes:
      - MongoDB (default, production): `cd calm-hub/deploy && docker-compose up` — starts Hub on port 8080 + MongoDB
      - NitriteDB (embedded, standalone): no MongoDB needed; single process. Good for local dev.
      Inline image: `![Hub namespace resource tree: root Hub node, two namespace branches (org.mybank, finos), each with architectures/patterns/standards/controls leaves with version numbers](../illustrations/exported/m03-hub-namespace-tree.svg)`

      ### Namespaces
      Organizational isolation: `org.mybank/payments` — all payment architectures. `org.mybank/trading` — trading architectures. `finos/reference` — FINOS reference patterns. Namespaces separate concerns and enable access control (who can publish to which namespace).

      ### REST API shape
      Key endpoints:
      - `GET /calm/namespaces` — list namespaces
      - `POST /calm/namespaces/{ns}/architectures` — publish architecture
      - `GET /calm/namespaces/{ns}/architectures` — list architectures in namespace
      - `GET /calm/namespaces/{ns}/patterns` — list patterns
      Same shape for standards, controls, decorators.
      Swagger UI at `http://localhost:8080/q/swagger-ui/` for interactive exploration.

      ### Auth profiles (CRITICAL PITFALL)
      Hub has three auth profiles:
      - `no-auth`: used by the docker-compose quickstart. All requests allowed. Use only locally.
      - `secure`: default Maven profile. Every request returns 401 without a valid token. NEVER use no-auth in production.
      - `proxy-auth`: for reverse proxy setups where auth is handled upstream.
      If you start Hub via Maven directly without `-Dquarkus.profile=no-auth` you get 401 on everything — this is expected behavior, not a bug.

      ### CLI hub commands
      All shipping in CLI 1.45.0 (confirmed — not a gap):
      Push: `calm hub push architecture my-arch.calm.json --calm-hub-url http://localhost:8080`
      Pull: `calm hub pull architecture --calm-hub-url http://localhost:8080`
      List: `calm hub list architectures --calm-hub-url http://localhost:8080`
      List patterns: `calm hub list patterns --calm-hub-url http://localhost:8080`
      List namespaces: `calm hub list namespaces --calm-hub-url http://localhost:8080`
      Configure default Hub URL: `calm init-config --calm-hub-url http://localhost:8080` → saves to ~/.calm.json so flag is not required on every command.

      ### Hub-integrated validate
      Use Hub-hosted patterns in validate without downloading: `calm validate -a arch.calm.json -p http://localhost:8080/calm/namespaces/myorg/patterns/secure-api --calm-hub-url http://localhost:8080`
      This is the production workflow: teams don't copy pattern files locally — they reference Hub URLs in their CI pipeline.

      ### Hub MCP endpoint
      Enable with `CALM_MCP_ENABLED=true` environment variable. Available at `http://localhost:8080/mcp` (Streamable HTTP transport). AI agents can query: "what patterns exist for secure API services in namespace org.mybank?" — Hub returns a list of patterns with names, versions, and descriptions. This is the foundation for AI-assisted architecture discovery.

      ### Hub local quickstart
      ```bash
      cd /path/to/architecture-as-code/calm-hub/deploy
      docker-compose up
      # Hub available at http://localhost:8080
      # Swagger UI at http://localhost:8080/q/swagger-ui/
      ```
      The docker-compose uses the no-auth profile — any request succeeds without authentication. Suitable for local learning only.

      ### The publish/consume workflow
      Step-by-step with inline image `![Architect pushes to Hub, Hub stores versioned artifact, product team pulls via CLI, CI validates against Hub-hosted pattern](../illustrations/exported/m03-hub-publish-consume-flow.svg)`:
      1. Architect creates/updates architecture locally (CLI or Studio)
      2. `calm hub push architecture v2.calm.json` → Hub stores version with history
      3. Product team references Hub artifact by namespace + version in CI
      4. CI runs: `calm validate -a arch.calm.json -p http://hub:8080/ns/patterns/secure-api`
      5. On PR merge: `calm hub push architecture v3.calm.json` → new version published
      Teams reference namespace + version; breaking changes require new version; old consumers continue working.

      ### Teaching note for Lab 3
      Lab 3 does NOT require a running Hub. Hub is taught conceptually in this chapter. The Lab 3 stretch goal optionally shows pushing to a local Hub Docker instance, but the core lab steps work entirely with local files and GitHub Actions.

    Content arc:
    - TL;DR (3-4 bullets): CALM Server exposes `calm validate` as an HTTP endpoint; platform teams run one central server, all teams POST architectures to validate (no per-team CLI install); POST /calm/validate → {hasErrors, hasWarnings}; built-in rate limiting at 100 req/15min per IP.
    - Why it matters: in large organizations, mandating CLI installation on every developer machine is impractical. CALM Server centralizes validation as a service — teams call an endpoint, platform team controls the schema version.
    - The concept H3 subsections:

      ### What CALM Server is
      A standalone HTTP server that exposes CALM validation as a REST endpoint. Bundled schemas included — no external dependencies. Platform teams run one instance; all dev teams POST their architectures to a shared validation API.
      Inline image: `![Platform validation pattern: Developer sends POST /calm/validate request to calm-server, server responds with hasErrors and hasWarnings; contrasted with local CLI validate flow](../illustrations/exported/m03-calm-server-platform-pattern.svg)`

      ### Starting the server
      ```bash
      calm-server                                # port 3000, localhost only
      calm-server --port 8080                    # custom port
      calm-server -s /path/to/custom-schemas     # custom schema directory
      calm-server -c http://hub:8080             # linked to CALM Hub for remote schemas
      ```
      Default binds to localhost only — prevents accidental exposure. For platform-wide deployment, place behind a reverse proxy with auth.

      ### Health check
      ```bash
      curl http://localhost:3000/health
      # {"status": "OK"}
      ```
      Use this in k8s liveness probes or load balancer health checks.

      ### The validate endpoint
      ```bash
      curl -X POST http://localhost:3000/calm/validate \
        -H "Content-Type: application/json" \
        -d @my-architecture.calm.json
      # Response: {"hasErrors": false, "hasWarnings": false, ...}
      ```
      Non-zero `hasErrors` indicates schema violations. Integrate into CI: `POST` architecture → parse response → fail pipeline if `hasErrors: true`.

      ### Rate limiting
      Built-in: 100 requests per 15-minute window per IP address. Configurable:
      - `--rate-limit-window <ms>` (default: 900000 — 15 minutes)
      - `--rate-limit-max <n>` (default: 100)
      For high-volume CI farms, increase the limit or use per-team IP allocation.

      ### Security note
      CALM Server has no built-in authentication. For enterprise use: bind to internal network only, place behind an authenticated reverse proxy (e.g., nginx + OAuth2 Proxy). Never expose CALM Server directly to the internet.

      ### CLI vs Server: when to use which
      | Need | Use |
      |------|-----|
