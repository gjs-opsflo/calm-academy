    **Chapter 3.1: CALM CLI (4 slides)**

    Slide 3: Chapter intro
    "Chapter 3.1: CALM CLI — Your Architecture Toolbox"
    Include the CLI command taxonomy image: `![CALM CLI command taxonomy](../illustrations/exported/m03-calm-cli-command-map.svg)`

    Slide 4: The validate command
    - `calm validate -a arch.calm.json` — schema only
    - `calm validate -a arch.calm.json -p pattern.json` — schema + pattern conformance
    - `-f pretty` (human) vs `-f junit` (CI dashboard)
    - Key pitfall: these are TWO different checks — run them as SEPARATE steps
    Speaker note: Emphasize the two-level validation insight — this is the most common mistake.

    Slide 5: Four more commands
    Quick reference table or bullets:
    - `calm generate -p pattern.json -o out.calm.json` — instantiate from pattern
    - `calm diff -a v1.calm.json -b v2.calm.json -f summary` — compare versions
    - `calm timeline -a v1.calm.json v2.calm.json -o timeline.json` — evolution synthesis
    - `calm hub push/pull/list` — registry interaction (ships in 1.45.0)
    Speaker note: Point out hub commands ship — they are NOT future gaps.

    Slide 6: CI integration
    ```yaml
    - name: Install CALM CLI
      run: npm install -g @finos/calm-cli
    - name: Validate architecture
      run: calm validate -a arch.calm.json -f junit -o results.xml
    ```
    "Architecture gets the same CI treatment as code."
    Speaker note: The -f junit flag is the CI integration point — CI dashboards parse JUnit XML natively.

    "Edit canvas or JSON — both stay in sync"
    Include: `![Bidirectional sync: drag node → JSON updates; edit JSON → canvas moves](../illustrations/exported/m03-bidirectional-sync.svg)`

    "Chapter 3.3: CALM Hub — The Architecture Registry"

    - Namespace: org.mybank/payments, org.mybank/trading, finos/reference

    - REST API: /calm/namespaces/{ns}/architectures, /patterns, /standards

    Slide 12: Publish/consume workflow
    Include: `![Hub publish-consume workflow: Architect pushes → Hub stores → Team pulls → CI validates](../illustrations/exported/m03-hub-publish-consume-flow.svg)`
    Speaker note: Version pinning is the key concept — teams reference namespace + version; breaking changes require a new version.

    Slide 13: Hub quickstart + MCP endpoint
    ```bash
    cd calm-hub/deploy && docker-compose up
    # Hub at http://localhost:8080
    # Swagger: http://localhost:8080/q/swagger-ui/
    ```

    "AI agents can query Hub: 'what patterns exist for secure API services?'"
    Speaker note: Critical warning — Hub defaults to secure profile (401 on everything). docker-compose uses no-auth.

    Slide 14: Chapter intro
    "Chapter 3.4: CALM Server — Validation as a Service"
    Include: `![CALM Server: platform team runs one server; all dev teams POST architectures to validate](../illustrations/exported/m03-calm-server-platform-pattern.svg)`

    Slide 15: API and use case
    ```bash
    calm-server --port 3000
    curl GET /health          # → {"status": "OK"}
    curl POST /calm/validate  # → {hasErrors: false, ...}
    ```
    Rate limiting: 100 req/15min (configurable)
    "Platform team controls schema version. No per-developer CLI install required."
    Speaker note: No built-in auth — bind to internal network, add reverse proxy for enterprise use.

    Quick list:
    1. Interactive Preview Panel — live diagram as you type

    **Chapter 3.6: Patterns and Standards (3 slides)**

    Slide 18: Chapter intro
    "Chapter 3.6: Patterns and Standards — The Blueprint System"
    Include: `![Pattern file with constraints via calm generate arrow produces architecture file](../illustrations/exported/m03-pattern-vs-architecture.svg)`

    Slide 19: Pattern vs architecture vs standard
    Table:
    | Artifact | Extension | Purpose | Validates with |
    |----------|-----------|---------|----------------|
    | Pattern | .pattern.json | Template | calm validate -p |
    | Architecture | .calm.json | Instance | calm validate -a |
    | Standard | varies | Org constraints | Referenced in -a validate |
    Speaker note: This distinction is the most important concept in this chapter.

    Slide 20: The calm generate workflow
    ```bash
    # Interactive
    calm generate -p secure-api-service.pattern.json -o my-api.calm.json

    # Non-interactive (CI/scripting)
    calm generate -p pattern.json --option-choices '{"api":"REST"}' -o arch.calm.json

    # From Hub
    calm generate -p http://hub:8080/ns/patterns/secure-api -o arch.calm.json
    ```
    "Generate + validate in sequence = your organization's architecture onboarding story."

    **Chapter 3.7: CI/CD Integration (5 slides)**

    Slide 21: Chapter intro — architecture as a pipeline stage
    "Chapter 3.7: CI/CD Integration — Architecture Gates"
    "Architecture drift starts the moment developers stop reading the docs. CI gates close that gap."
    Include: `![GitHub Actions flow: PR opened → validate → pass/fail branches → merge or block](../illustrations/exported/m03-cicd-gate-flow.svg)`

    Slide 22: The GitHub Actions workflow
    Show the key YAML snippet (abbreviated — just the CALM steps):
    ```yaml
    - name: Install CALM CLI
      run: npm install -g @finos/calm-cli
    - name: Validate schema
      run: calm validate -a architectures/secure-api.calm.json -f pretty
    - name: Validate pattern conformance
      run: calm validate -a architectures/secure-api.calm.json -p patterns/... -f pretty
    - name: JUnit report
      run: calm validate -a ... -f junit -o results.xml
    ```
    Speaker note: Two separate validate steps by design — schema errors and pattern errors need separate feedback.

    Slide 23: calm diff as PR automation
    - `calm diff -a before.calm.json -b after.calm.json -f summary`
    - `--exit-code`: exits non-zero WHEN CHANGES ARE DETECTED
    - Correct use: diff PR version vs Hub-published baseline (not vs previous commit)
    - Wrong use: `--exit-code` on every commit will block every legitimate architecture update
    Speaker note: This pitfall is where half the class goes wrong. Be explicit.

    Slide 24: Hub-integrated pipeline
    Include: `![Full pipeline: PR → validate → diff → tests → merge → Hub push → CALM Guard → Deploy](../illustrations/exported/m03-full-stack-pipeline.svg)`
    Speaker note: The Hub push step on merge is the key architectural commitment. After merge, the architecture is the source of truth in Hub.

    Slide 25: The blocking strategy
    "Architecture must pass before IaC applies."
    ```
    calm validate (gate 1) →
    terraform plan (gate 2) →
    calm hub push (publish) →
    CALM Guard (continuous compliance)
    ```
    "Architecture is code. It merges, it deploys, it gets compliance-checked."
