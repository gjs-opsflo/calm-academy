```yaml
name: CALM Architecture Gate
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
jobs:
  validate-architecture:
    name: Validate CALM Architecture
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
      - name: Install CALM CLI
        run: npm install -g @finos/calm-cli
      - name: Validate architecture against schema
        run: calm validate -a architectures/secure-api.calm.json -f pretty
      - name: Validate architecture against pattern
        run: |
          calm validate \
            -a architectures/secure-api.calm.json \
            -p patterns/secure-api-service.pattern.json \
            -f pretty
      - name: Output JUnit test report
        run: |
          calm validate \
            -a architectures/secure-api.calm.json \
            -p patterns/secure-api-service.pattern.json \
            -f junit \
            -o calm-validation-results.xml
      - name: Upload test results
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: calm-validation-results
          path: calm-validation-results.xml
```

    Content arc:
    - TL;DR (4 bullets): Extension is "CALM Tools" by FINOS (search by name in VS Code marketplace); preview panel shows live diagram as you type; real-time validation in the Problems panel catches errors before CI; timeline navigation lets you explore architecture evolution by milestone.
    - Why it matters: the extension brings the validate-and-visualize loop into the developer's natural environment — no context switching to a browser or terminal for every edit.
    - The concept H3 subsections:

      ### Installing the extension
      - Search "CALM Tools" in VSCode Extensions sidebar (publisher: FINOS)
      - OR: `code --install-extension FINOS.calm-vscode-plugin`
      - OR: Marketplace URL (VS Code marketplace, extension ID FINOS.calm-vscode-plugin)
      - Version at time of writing: 0.6.0
      - Schema detection: extension automatically activates for any JSON file with a `$schema` field pointing to the CALM schema URL.

      ### Interactive Preview Panel
      Opens automatically when editing a `.calm.json` file. Shows a live diagram of the architecture — nodes and relationships update as you type. Click any node or relationship to inspect its properties. Click "Auto Layout" to re-run ELK hierarchical layout. Unlike CALM Studio (which is browser-based), the Preview Panel is embedded in the editor — no context switch.

      ### Tree View Navigation
      Available in the VS Code sidebar (CALM icon). Provides a tree view of: Nodes (grouped by node-type), Relationships (grouped by relationship-type), Flows (if defined). Search and filter nodes by name or type. Click to navigate to the node's definition in the JSON file.

      ### Timeline Navigation
      If the architecture file includes a `timeline` section with milestones, the extension provides a Timeline panel showing each milestone. Click a milestone to open a snapshot of the architecture at that point in time. Useful for reviewing how an architecture evolved.

      ### Real-Time Validation (Problems Panel)
      The extension validates your architecture on open, save, and file switch. Validation errors appear in VS Code's Problems panel (View → Problems or Cmd/Ctrl+Shift+M). Click any error to jump to the offending line in the JSON. The extension uses bundled CALM 1.2 schemas — no network access required. This means validation works offline and in air-gapped environments.

      ### Template/Documentation Mode
      The extension includes a live `docify` mode: open the documentation preview to see auto-generated HTML/Markdown from the architecture, updated in real-time. Useful for reviewing what `calm docify` will produce before running it.
      Inline image: `![VSCode window with CALM architecture JSON in center editor, Preview Panel diagram on right, Tree View on left sidebar, Problems panel at bottom](../illustrations/exported/m03-vscode-extension-panels.svg)`

      ### Configuration (.vscode/settings.json)
      ```json
      {
        "calm.urlMapping": "calm-mapping.json",
        "calm.schemas.additionalFolders": ["./my-schemas"],
        "calm.files.globs": ["calm/**/*.json", "calm/**/*.y?(a)ml"]
      }
      ```
      - `calm.urlMapping`: maps remote schema URLs to local files (for offline/air-gapped use)
      - `calm.schemas.additionalFolders`: adds extra schema directories
      - `calm.files.globs`: controls which files the extension activates on (default: JSON/YAML with $schema)

      ### When the extension catches errors before CI
      The preventive validation story: the Problems panel catches schema errors as you type, before you commit. This is significantly cheaper than a CI failure — no wait time, no PR round-trip. The extension and `calm validate` use the same schema, so a file that passes the Problems panel will pass CI validation.

    - Common mistakes (3): not realizing the extension auto-activates on `$schema` detection; confusing the Preview Panel with CALM Studio (extension is embedded; Studio is browser-based); expecting `calm.files.globs` to enable the extension for non-JSON files without the $schema field.

    Content arc:
    - TL;DR (4-5 bullets): validate on every PR — break the merge if architecture is invalid; `-f junit` produces parseable CI test reports; `calm diff --exit-code` gates on unexpected architecture changes (not all changes); Hub-integrated pipeline: validate against Hub-hosted patterns, publish to Hub on merge; Lab 3 implements this end-to-end.
    - Why it matters: architecture standards only matter if they're enforced. Docs are ignored; CI gates are not.
    - The concept H3 subsections:

      ### Why architecture validation in CI
      Architecture drift — the gap between the documented architecture and what's actually deployed — starts the moment developers start ignoring the diagram. CI/CD enforcement closes this gap: if the architecture doesn't pass validation, the branch doesn't merge. "Treat architecture like a test" — it fails fast, close to the developer, before production.

      ### The basic GitHub Actions workflow
      Present the EXACT YAML from RESEARCH.md "Lab 3: CI/CD Gate — Design" section. Every flag must match what was verified against CLI 1.45.0. Show the complete workflow with all four steps: checkout → setup-node → install CALM CLI → validate schema → validate pattern → JUnit output → upload artifact.
      Inline image: `![GitHub Actions flow: PR opened → validate step (green/red) → junit output → merge allowed or blocked](../illustrations/exported/m03-cicd-gate-flow.svg)`

      ### JUnit XML output and CI dashboards
      `-f junit` produces JUnit XML compatible with GitHub Actions test reporter, Jenkins, CircleCI, etc. The `upload-artifact` step saves results regardless of outcome (`if: always()`). Most CI platforms auto-parse JUnit XML when uploaded — makes validation failures visible in the PR interface as test failures, not just as "step failed" messages.
      Show: `calm validate -a arch.calm.json -p pattern.json -f junit -o results.xml`

      ### Two validation levels in one pipeline
      CRITICAL from Pitfall 4: always run schema validation AND pattern validation as SEPARATE steps:
      Step 1: `calm validate -a arch.calm.json -f pretty` — schema-only (tells engineer if the file is malformed CALM)
      Step 2: `calm validate -a arch.calm.json -p pattern.json -f pretty` — pattern conformance (tells engineer if the architecture doesn't meet the org's requirements)
      Running them separately gives engineers better error messages. A schema error means "fix your JSON structure." A pattern error means "fix your topology/content."

      ### calm diff as PR review automation
      `calm diff -a before.calm.json -b after.calm.json -f summary` — shows what changed between versions in a human-readable summary.
      `--exit-code` flag: exits non-zero when CHANGES ARE DETECTED.
      CRITICAL from Pitfall 6: explain correct use of `--exit-code`. It is NOT "block the build if anything changed." It IS "block if the architecture differs from what's expected." The correct use:
      - `calm diff --exit-code -a hub-published-version.calm.json -b pr-version.calm.json` — ensures the PR doesn't accidentally modify an architecture without intent. If the PR intentionally updates the architecture, the Hub-published version becomes the new baseline on merge.
      - Using `--exit-code` on every commit without context will fail every PR that contains a legitimate architecture update.

      ### Installing CALM CLI in CI
      Standard pattern: `npm install -g @finos/calm-cli` in the CI step. Alternatively, pin a version: `npm install -g @finos/calm-cli@1.45.0` to prevent unexpected behavior from automatic upgrades.
      For Docker-based CI: add to your builder image: `RUN npm install -g @finos/calm-cli`.

      ### Hub-integrated pipeline
      When Hub is deployed, the pipeline becomes:
      1. PR opened → `calm validate -a arch.calm.json -p http://hub:8080/ns/patterns/secure-api` (pattern pulled from Hub at validate time — no local pattern file needed)
      2. `calm diff -a (Hub-published v1) -b (PR version) -f summary` (shows change summary in PR)
      3. Tests pass → PR merged
      4. `calm hub push architecture arch.calm.json --calm-hub-url http://hub:8080` (new version published to Hub)
      5. CALM Guard picks up new version from Hub, runs compliance check
      6. Terraform/IaC applies (infra conforms to the now-validated architecture)

      ### The full stack pipeline
      Inline image: `![Full pipeline: PR opened → calm validate → calm diff → tests pass → merge → Hub push → CALM Guard compliance → Terraform/IaC apply → Deploy](../illustrations/exported/m03-full-stack-pipeline.svg)`
      The mental model: architecture is code. It gets the same pipeline treatment as application code. Architecture that fails CI doesn't merge. Architecture that merges becomes the versioned source of truth in Hub. Guard continuously verifies that running systems match what's in Hub.

      ### Blocking strategy
      "Architecture must pass before IaC applies." This is the architectural compliance gate. Even if `terraform plan` succeeds, the pipeline should block if the architecture hasn't been validated. The sequence: calm validate → (if pass) terraform plan → terraform apply. This catches the case where a developer updates infrastructure without updating the architecture doc.

You fork a starter repo, complete the CI workflow stub, and push both a valid and an invalid architecture to watch the gate work."
    - Common mistakes (5): schema vs pattern validation as one step; `--exit-code` on every commit blocking all architecture updates; not using `-f junit` in CI (human output is not parseable by test reporters); skipping `upload-artifact` (no test report stored on failure); pinning Hub URL directly in architecture files instead of via `--calm-hub-url` flag.
