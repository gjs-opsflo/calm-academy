    - Why it matters: patterns are how architecture standards propagate at scale — instead of telling teams "build it this way," you give them a pattern and enforce conformance in CI.
    - The concept H3 subsections:

      ### What a pattern is
      A CALM Pattern is a JSON Schema document that defines the TEMPLATE for a conforming architecture. It uses JSON Schema constraints (`const`, `required`, `properties`) to enforce:
      - Which node types must be present
      - Which relationships must exist
      - What metadata fields are required
      `calm validate -p pattern.json` validates the pattern itself. `calm generate -p pattern.json` instantiates an architecture from it.

      ### Pattern vs architecture vs standard
      Table:
      | Artifact | Extension | Validates with | Purpose |
      |----------|-----------|----------------|---------|
      | Pattern | `.pattern.json` | `calm validate -p` | Template defining required topology |
      | Architecture | `.calm.json` | `calm validate -a` | Conforming instance of zero or more patterns |
      | Standard | (varies) | Referenced in validate call | Org-specific extensions on top of the CALM spec |
      Inline image: `![Pattern file on the left with const constraints flowing via calm generate arrow to architecture file on the right with values filled in](../illustrations/exported/m03-pattern-vs-architecture.svg)`
