    ### Why the AI doesn't invent node types
    The central claim of the AI path is that it produces valid CALM — meaning correct node types, correct relationship syntax, correct schema structure. Explain the mechanism:

    ### When the AI path breaks down
    Be honest about limitations. Cover 4-5 scenarios where AI-generated CALM needs human review:

    1. **Ambiguous descriptions**: "The system has a caching layer" — is the cache a `service`, a `database`, or a `network` node? The AI makes a reasonable guess; a CALM-educated architect makes a principled choice. (Teaser: Module 2 teaches which types fit which real-world components.)

    2. **Missing nodes**: A 3-sentence description of a "web app with an API and database" omits the load balancer, the identity provider, the CDN. AI generates what you describe. If your description is incomplete, your CALM is incomplete.

    3. **Incorrect relationship direction**: `connects` has a source and destination — the direction implies data flow. AI may flip source and destination in ambiguous cases. A human reviewer catches this; an LLM might not.

    4. **Relationship type selection**: `connects` vs. `interacts` vs. `deployed-in` each carry semantic meaning for automated analysis tools (CALM Guard, CALM Studio). The AI usually picks correctly for simple descriptions; complex topologies need human judgment.

    5. **Edge cases in the spec**: CALM 1.2 has features the AI path doesn't surface in Module 0 — interfaces, controls, decorators, timelines, `options` (architectural decision branches). These require Module 2+ knowledge to use correctly.
